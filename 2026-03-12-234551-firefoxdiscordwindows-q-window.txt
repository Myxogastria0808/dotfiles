  根本原因の分析

  現在の環境

  ユーザーのセッション構成：
  - defaultSession = "plasmax11" — SDDMのデフォルトはKDE Plasma X11
  - Hyprlandも home/config/hyprland/default.nix で設定済みで、SDDM
  からの選択可能

  つまり、2つの原因シナリオが考えられます。

  ---
  シナリオA: Hyprlandを使用している場合（可能性が高い）

  home/config/hyprland/default.nix
  の54〜81行目で、以下の3つの環境変数がコメントアウトされている：

  # "MOZ_ENABLE_WAYLAND,1"          ← Firefox に必要
  # "ELECTRON_OZONE_PLATFORM_HINT,auto"  ← Discord/Electron に必要
  # "NIXOS_OZONE_WL,1"              ← NixOS Electronラッパー用

  これが両方の問題の根本原因です：

  フリーズの原因:
  - Firefox と Discord が XWayland（X11互換レイヤー）経由で動作
  - fcitx5 の waylandFrontend = true
  がXWayland上のウィンドウにフォーカスした際に競合しフリーズ
  - XWayland上のElectron(Discord)はGPUレンダリングで不安定になりやすい

  Super+Qが効かない原因:
  - XWayland上で動作するX11アプリはSuperキーなどのグローバルショートカットを横取
  りできる
  - FirefoxやDiscordがフォーカスを持っている間、HyprlandがSuper+Qを受け取れない
  - これはXWaylandの既知の問題：Hyprland Wiki - XWayland

  ---
  シナリオB: KDE Plasma X11を使用している場合

  - defaultSession = "plasmax11" のままSDDMから起動している場合
  - KDE PlasmaにはSuper+Qの「ウィンドウを閉じる」バインドがデフォルトで存在しな
  い（Alt+F4が標準）
  - Firefox/DiscordのフリーズはX11上のfcitx5との競合が原因の可能性

  ---
  現在のdotfilesで起きている問題（複数重複）

  問題1: systemd.enable がUWSMと競合している（最有力）

  home-managerのHyprlandモジュールはデフォルトでsystemdインテグレーションが有効
  になっています。しかしwithUWSM = 
  trueも有効にしているため、2つのsystemdセッション管理が衝突します。

  "If you use the Home Manager module, make sure to disable systemd integration,
   as it conflicts with UWSM."
  — Hyprland on Home Manager Wiki

  現在の home/config/hyprland/default.nix に systemd.enable = false
  が設定されていないので、この衝突が発生しています。

  問題2: UWSMではenvセクションの環境変数が子プロセスに伝播しない

  昨日追加した MOZ_ENABLE_WAYLAND,1 等の修正は実は効果がない可能性が高いです。

  "UWSM users should avoid placing environment variables in hyprland.conf. 
  Instead use ~/.config/uwsm/env"
  — Hyprland Wiki: Environment variables
  — Arch Wiki: Hyprland

  また home-manager の home.sessionVariables
  もUWSMセッションには伝わらないというバグが報告されています：home-manager issue
   #7209

  問題3: Hyprlandフレークとシステムのmesaバージョン不一致

  flake から最新のHyprlandを使っている一方、システムのmesa（nixpkgs-unstable）と
  のバージョンがずれるとGPUレンダリングが不安定になります。

  "If you start experiencing lag and FPS drops... when using the Hyprland flake,
   it is most likely a mesa version mismatch"
  — Hyprland on NixOS Wiki

  問題4: Intel iGPU固有のHyprlandバグ

  Intel UHD
  620でHyprlandを使うと高GPU使用率・フリーズが起きる既知の問題があります（issue 
  #5441, issue #12106）。参照dotfilesのNVIDIA
  GPUではこのコードパスは通りません。

  1. NVIDIAだから — Intel
  iGPU固有のHyprlandバグ（高GPU使用率→フリーズ）が存在しない
  2. Hyprlandフレークを使っていないから —
  nixpkgsのHyprlandを使うのでmesaバージョン不一致が起きない
  3. Firefoxが入っていないから —
  XWayland上のFirefoxはフリーズの最大の引き金だが参照dotfilesには存在しない

  現在のdotfilesの複合的な問題：

❯ zen kernelって何？ 

● Linux カーネルのカスタムビルドの一つです。

  通常のLinuxカーネルは「サーバー・デスクトップ・組み込みすべてに対応」という汎
  用設計ですが、zen kernel 
  はデスクトップ・ゲーム用途の応答性を最優先にチューニングしています。

  主な違い：

  ┌─────────────────────┬────────────────────────┬──────────────────────────┐
  │        項目         │      通常カーネル      │        zen kernel        │
  ├─────────────────────┼────────────────────────┼──────────────────────────┤
  │ CPUスケジューラの応 │ 汎用設定               │ 低レイテンシ優先         │
  │ 答性                │                        │                          │
  ├─────────────────────┼────────────────────────┼──────────────────────────┤
  │ タイマー精度        │ 250Hz                  │ 1000Hz（より細かく割り込 │
  │                     │                        │ み）                     │
  ├─────────────────────┼────────────────────────┼──────────────────────────┤
  │ プリエンプション    │ 必要時のみ             │ 積極的（UI操作を優先）   │
  ├─────────────────────┼────────────────────────┼──────────────────────────┤
  │ 対象                │ サーバー〜デスクトップ │ デスクトップ・ゲーム特化 │
  │                     │ 全般                   │                          │
  └─────────────────────┴────────────────────────┴──────────────────────────┘

  体感できる効果：
  - アプリ操作やウィンドウ切り替えがスムーズになる
  - バックグラウンドで重い処理をしていてもUIがカクつきにくい
  - HyprlandのようなWaylandコンポジターと相性が良い

  トレードオフ：
  - スループット（サーバー的な大量データ処理）はわずかに落ちる可能性がある
  - バッテリー消費が若干増える場合がある（タイマーが細かくなるため）

  ThinkPadのような「デスクトップ用途のノートPC」には基本的にメリットの方が大きい
  です。
