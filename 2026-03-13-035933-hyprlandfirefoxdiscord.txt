  ---
  原因分析

  1. 最有力: xdg-desktop-portal-gtk の欠如

  modules/display-manager/default.nix:28-30 でコメントアウトされています:

  # extraPortals = with pkgs; [
  #   xdg-desktop-portal-gtk  # Add this on non-KDE/GNOME desktops
  # ];

  Hyprland では xdg-desktop-portal-hyprland
  だけでは不十分で、多くのアプリ（Firefox、Discord、GTKアプリ全般）が起動時に
  org.freedesktop.portal.FileChooser
  等のGTKポータルインターフェースに接続しようとしてタイムアウト（通常
  5〜30秒）を待ちます。これが「全アプリの起動が遅い」の典型的な原因です。

  2. Hyprland Wiki 推奨の portal.config 未設定

  xdg-desktop-portal 1.18+
  では、どのポータルがどのインターフェースを担当するかを明示しないと resolver
  がタイムアウトします。

● XDPH とは

  XDPH = xdg-desktop-portal-hyprland の略称です。

  xdg-desktop-portal は「アプリ ↔ デスクトップ環境」の橋渡しをする D-Bus
  サービスですが、インターフェースごとに実装が分かれています:

  ┌─────────────────────────────┬───────────┬────────────────────────┐
  │      インターフェース       │   XDPH    │ xdg-desktop-portal-gtk │
  ├─────────────────────────────┼───────────┼────────────────────────┤
  │ ScreenCopy（画面共有）      │ ✅ 対応   │ ❌                     │
  ├─────────────────────────────┼───────────┼────────────────────────┤
  │ GlobalShortcuts             │ ✅ 対応   │ ❌                     │
  ├─────────────────────────────┼───────────┼────────────────────────┤
  │ Screenshot                  │ ✅ 対応   │ ❌                     │
  ├─────────────────────────────┼───────────┼────────────────────────┤
  │ FileChooser（ファイル選択） │ ❌ 未実装 │ ✅ 対応                │
  ├─────────────────────────────┼───────────┼────────────────────────┤
  │ OpenURI                     │ ❌ 未実装 │ ✅ 対応                │
  ├─────────────────────────────┼───────────┼────────────────────────┤
  │ Settings（テーマ情報）      │ ❌ 未実装 │ ✅ 対応                │
  └─────────────────────────────┴───────────┴────────────────────────┘

  なぜ gtk が必要か

  XDPH は Hyprland 固有の機能（画面共有など）しか実装していません。FileChooser（
  ファイルダイアログ）は XDPH に実装がないため、アプリがそれをリクエストすると:

  アプリ → D-Bus → XDPH に FileChooser を要求
                  → XDPH: 未実装 → タイムアウト待機 (5〜30秒)

  xdg-desktop-portal-gtk を追加すると:

  アプリ → D-Bus → XDPH: ScreenCopy 等は自分が処理
                → gtk:  FileChooser 等は自分が処理

  これが今回の修正で xdg-desktop-portal-gtk を extraPortals に追加した理由です。

❯ config.common.default = "*" の必要性がよくわかっていない 

● config.common.default = "*" の仕組み

  xdg-desktop-portal 1.18 で「portal resolver」という仕組みが導入されました。

  resolver の動作

  アプリが FileChooser をリクエストしたとき、resolver
  はどのポータル実装を使うかを決める設定ファイルを探します。

  設定がない場合（デフォルト）:

  1. XDG_CURRENT_DESKTOP=Hyprland を見る
  2. /usr/share/xdg-desktop-portal/portals/Hyprland.portal を探す
  3. XDPH の .portal ファイルには FileChooser が書かれていない
  4. 次の候補を探す... タイムアウト待機

  config.common.default = "*" があると:

  1. 設定を見る → "どの DE でも * (全部自動)" と書いてある
  2. FileChooser を実装しているポータルを直接選ぶ → gtk が応答
  3. 即座に解決

  では実際に必要か？

  NixOS で programs.hyprland.enable = true にすると、NixOS の Hyprland
  モジュールが自動で以下を生成します:

  # /etc/xdg/xdg-desktop-portal/hyprland-portals.conf
  [preferred]
  default=hyprland;gtk

  これが XDG_CURRENT_DESKTOP=Hyprland のときに自動適用されるので、UWSM + NixOS 
  Hyprland モジュール使用中は config.common.default = "*" 
  はなくても動く可能性が高いです。

  結論

  ┌─────────────────────────────────────────┬────────────────────────────────┐
  │                  状況                   │             必要性             │
  ├─────────────────────────────────────────┼────────────────────────────────┤
  │ NixOS programs.hyprland.enable = true + │ おそらく不要（自動生成される） │
  │  UWSM                                   │                                │
  ├─────────────────────────────────────────┼────────────────────────────────┤
  │ 手動インストール・非NixOS               │ 必要                           │
  ├─────────────────────────────────────────┼────────────────────────────────┤
  │ 明示的に動作を保証したい                │ 保険として有効                 │
  └─────────────────────────────────────────┴────────────────────────────────┘
