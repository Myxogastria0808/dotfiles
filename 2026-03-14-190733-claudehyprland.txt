  ---
  あなたの構成で該当する可能性が高いバグ

  最有力: XDG_CURRENT_DESKTOP=start-hyprland 問題

  nixpkgsの withUWSM = true 設定で、XDG_CURRENT_DESKTOP が Hyprland ではなく
  start-hyprland にセットされてしまうバグ。これにより起動時にHyprlandが警告を出
  し、初期化が遅延する。

  - nixpkgs PR #478472 で2026年1月に修正済み
  - ただしフラーク版（あなたの構成）は影響範囲が異なる可能性あり

  確認方法：
  # ログイン後に実行
  echo $XDG_CURRENT_DESKTOP
  # "start-hyprland" と出たらこのバグ

  ---
  回避策

  方法1: DesktopNames を明示的に指定（nixpkgsモジュールのバグ回避）

  modules/display-manager/hyprland/default.nix に追加：

  # UWSM session entry の XDG_CURRENT_DESKTOP を強制上書き
  environment.sessionVariables.XDG_CURRENT_DESKTOP = "Hyprland";

  方法2: サービスの起動順序レースコンディション修正

  UWSMとhome-managerの組み合わせで、各サービスが起動前のコンポジターに接続しよう
  としてタイムアウト待ちが発生するバグ。

  home/config/hyprland/default.nix に追加：

  # xdg-desktop-portal が Hyprland ソケット準備前に起動するのを防ぐ
  systemd.user.services.xdg-desktop-portal-hyprland = {
    Unit.After = lib.mkForce [ "graphical-session.target" ];
  };

  ---
  実際に遅い箇所の特定

  まずログで原因を絞り込むと確実です：

  # ログイン後に実行して、どのサービスが遅いか確認
  systemd-analyze --user blame | head -20

  結果を共有してもらえれば、どのバグに当たるか特定できます。

● ボトルネックは明確です。

  3.827s wayland-session-waitenv.service   ← UWSMが環境変数を待っている
  3.570s wayland-wm@hyprland.desktop.service  ← Hyprland本体の起動

  wayland-session-waitenv.service がHyprlandの起動完了を待ち、WAYLAND_DISPLAY 等
  の環境変数がsystemdセッションに伝播されるのをタイムアウトまで待機しています。

  原因として、exec-once に uwsm finalize が抜けている可能性が高いです。UWSMはこ
  のコマンドでセッション準備完了を検知します。

  まず全体の依存チェーンを確認：

  systemd-analyze --user critical-chain wayland-session-waitenv.service

  結果を見せてもらえれば確定できますが、おそらく回避策はこれです。

  home/config/hyprland/default.nix の exec-once に追加：

  exec-once = [
    "uwsm finalize"   # ← これを追加：UWSMにセッション完了を通知
    "swww-daemon"
    "wl-paste --type text --watch cliphist store"
    "wl-paste --type image --watch cliphist store"
    "waybar"
  ];

  uwsm finalize は WAYLAND_DISPLAY、HYPRLAND_INSTANCE_SIGNATURE、DISPLAY 等を
  systemctl --user import-environment 経由で伝播させ、wayland-session-waitenv
  のタイムアウト待ちを解消します。
