● 差分の全体を把握しました。分析結果をまとめます。

  TTYに入れない原因について

  「Hyprland only」は関係ないと思います。TTYはSDDMやHyprlandよりずっと前（system
  dのgetty.target）で起動するので、DEの設定は無関係です。

  主な変更点（git diff より）

  旧 virtualisation.nix → 新 docker.nix / incus.nix / kvm.nix
  に分割されましたが、内容は実質同じです。ただし一点：

  # 旧 virtualisation.nix にあったが新設定では削除された
  - users.extraGroups.vboxusers.members = [ username "docker" "libvirtd" ]

  これは起動に影響しません。

  TTYに入れない本当の原因候補

  1. カーネルバージョン変更（最有力）

  boot.kernelPackages = pkgs.linuxPackages_latest + nixpkgs-unstable
  の組み合わせで、nixpkgs更新のたびに新しいカーネルを引いています。Intel GPU の
  DRM/KMS モジュールに regression があると、フレームバッファが機能せず TTY
  も真っ黒になります。

  2. scx_bpfland と新カーネルの相性

  カーネルが変わると eBPF の挙動も変わることがあります。

  診断手順

  systemd-boot で e キーを押し、カーネルパラメーターを編集：

  # 末尾に追加
  nomodeset

  nomodeset で TTY が見えるようになれば → KMS/Intel GPU の問題確定

  # また別の試みとして
  systemd.unit=rescue.target

  rescue mode で起動できれば、journalctl で原因を特定できます。

  SSH で別デバイスから接続（services.openssh.enable = true が有効なので）：

  ネットワーク接続さえ生きていれば SSH で入れる可能性があります。

  ssh hello@<IPアドレス>
  sudo journalctl -b 0 --no-pager | tail -50

  ---
  まず nomodeset を試してみてください。TTYが見えるようになった場合はKMSの問題な
  ので、その後の対処法を提案します。
